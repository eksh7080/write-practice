# Git 커밋 & 푸시 스킬

아래 단계를 순서대로 진행합니다.

---

## 1단계: 현재 상태 수집

아래 명령어를 실행해 현재 Git 상태를 파악합니다:

```bash
git branch --show-current
git status
git diff --stat
```

결과를 사용자에게 출력합니다:
- 현재 브랜치명
- 커밋되지 않은 파일 목록 (Untracked / Modified / Staged)
- 변경 통계 (삽입/삭제 라인 수)

---

## 2단계: 변경 사항 요약 표 출력

아래 명령어로 정보를 수집합니다:

```bash
git branch --show-current
git diff --name-only HEAD
git diff --shortstat HEAD
git status --short
```

수집한 정보를 아래 형식의 **마크다운 표**로 출력합니다:

| 항목 | 내용 |
|------|------|
| 현재 브랜치 | `브랜치명` |
| 변경된 파일 | `파일1`, `파일2`, ... |
| 변경 라인 수 | +N줄 추가, -N줄 삭제 |

변경된 파일이 하나도 없으면 "변경 사항이 없습니다."를 출력하고 종료합니다.

---

## 3단계: 커밋 메시지 입력 받기

AskUserQuestion 도구를 사용해 사용자에게 커밋 메시지를 입력받습니다.

질문: "커밋 메시지를 입력하세요. (비워두면 자동 생성)"

---

## 4단계: 커밋 메시지 처리

### 커밋 메시지를 입력한 경우 — Conventional Commits 검증

입력된 메시지가 아래 패턴에 맞는지 확인합니다:

```
<type>(<scope>): <description>
```

유효한 type 목록: `feat`, `fix`, `refactor`, `test`, `chore`, `docs`, `style`, `perf`, `ci`, `build`, `revert`

**형식이 올바른 경우**: 그대로 사용합니다.

**형식이 올바르지 않은 경우**: 아래 기준으로 자동 보정된 메시지를 제안하고, 사용자에게 수락 여부를 묻습니다.

보정 규칙:
- type이 없거나 잘못된 경우 → 변경된 파일 특성에 맞는 type으로 자동 추가
- scope가 없는 경우 → 주요 변경 파일명 또는 디렉토리를 scope로 추가
- description이 한국어가 아닌 경우 → 그대로 유지 (언어 변환 강제 안 함)

### 커밋 메시지를 입력하지 않은 경우 — 자동 생성

변경된 파일 목록과 diff를 분석해 아래 형식으로 커밋 메시지를 자동 생성합니다:

```
type(scope): 한국어 설명
```

**type 판단 기준:**

| 조건 | type |
|------|------|
| 새 파일 추가 (Untracked) | `feat` |
| 기존 파일 수정, 버그 수정 | `fix` |
| 기존 파일 리팩토링 | `refactor` |
| `*.test.*`, `*.spec.*` 파일 | `test` |
| `*.config.*`, `.env`, 빌드 파일 | `chore` |
| `*.md`, `*.txt`, 문서 파일 | `docs` |

**scope 판단 기준:**
- 변경 파일이 1개 → 해당 파일명 (확장자 제외)
- 변경 파일이 여러 개, 같은 디렉토리 → 디렉토리명
- 변경 파일이 여러 개, 다른 디렉토리 → 가장 핵심 파일의 디렉토리명

자동 생성된 메시지를 사용자에게 먼저 출력한 뒤, 아래 5단계 확인 절차로 넘어갑니다.

---

## 5단계: 커밋 메시지 최종 확인

4단계에서 확정된 커밋 메시지를 사용자에게 다시 보여준 뒤, AskUserQuestion 도구로 아래 3가지 선택지를 제공합니다.

```
확정된 커밋 메시지: "<커밋 메시지>"

어떻게 진행할까요?
```

**선택지:**
1. 현재 작성된 커밋메시지로 진행할까요?
2. 다른 커밋 메시지를 작성하시겠습니까?
3. 취소

### 선택 1 — 현재 메시지로 진행

확정된 커밋 메시지를 그대로 사용해 6단계로 이동합니다.

### 선택 2 — 다른 커밋 메시지 작성

AskUserQuestion 도구로 새 커밋 메시지를 다시 입력받습니다.

질문: "새로운 커밋 메시지를 입력하세요."

입력받은 메시지를 4단계의 Conventional Commits 검증 로직으로 다시 처리합니다.
검증/보정이 끝나면 5단계(최종 확인)로 돌아와 선택지를 다시 제공합니다.

### 선택 3 — 취소

아래 메시지를 출력하고 스킬을 종료합니다:

```
커밋이 취소되었습니다. 변경사항은 그대로 유지됩니다.
```

---

## 6단계: 커밋 및 푸시 실행

5단계에서 선택 1을 확정한 경우에만 아래 명령어를 순서대로 실행합니다.

**스테이징:**
```bash
git add -A
```
성공 시 "스테이징 완료" 출력, 실패 시 오류 메시지 출력 후 종료.

**커밋:**
```bash
git commit -m "<확정된 커밋 메시지>"
```
성공 시 커밋 해시와 함께 "커밋 완료" 출력, 실패 시 오류 메시지 출력 후 종료.

**푸시:**
```bash
git push origin <현재 브랜치명>
```
성공 시 아래 형식으로 결과를 출력합니다:

```
푸시 완료!
브랜치: <브랜치명>
원격 저장소: <remote URL>
커밋 메시지: <커밋 메시지>
```

실패 시 오류 메시지를 출력하고, upstream이 설정되지 않은 경우 아래 명령어를 제안합니다:

```bash
git push --set-upstream origin <현재 브랜치명>
```

사용자가 수락하면 해당 명령어를 실행합니다.
